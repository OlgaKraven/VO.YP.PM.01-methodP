1. Принципы системного и объектно‑ориентированного проектирования

1.1. Системный подход

Системное проектирование рассматривает информационную систему как совокупность взаимосвязанных элементов (данных, процессов, людей, оборудования, программ), объединённых общей целью.

Ниже — сводка ключевых принципов с краткими примерами из учебного проекта «Театр онлайн».

Принципы системного подхода

ПринципСодержаниеПример
ЦелостностьСистема рассматривается как единое целое, а не сумма частейИС «Театр онлайн» объединяет сайт, БД и админ‑панель
ИерархичностьКаждый уровень системы может рассматриваться как подсистемаМодуль «Билеты» внутри системы «Театр»
МодульностьСистема разбивается на независимые модулиОтдельные сервисы: пользователи, репертуар, оплата
УправляемостьДолжны быть механизмы контроля и регулированияЛогирование, отчёты, мониторинг
Обратная связьРезультаты влияют на последующие решенияАналитика продаж корректирует репертуар
РазвитиеВозможность эволюции и масштабированияДобавление онлайн‑трансляций без переписывания кода

1.2. Объектно‑ориентированное проектирование (ООП)

ООП описывает систему через объекты — сущности, обладающие состоянием (данными) и поведением (методами).

ПринципСутьПример
ИнкапсуляцияСокрытие внутреннего состояния объекта, доступ через методыUser скрывает пароль и хранит его хэш
НаследованиеСоздание подклассов на основе базового классаAdmin наследует User
ПолиморфизмОбщий интерфейс для разных реализацийdisplay() по‑разному для Ticket и Order
АбстракцияВыделение существенных характеристикPayment хранит сумму и статус, а не детали транзакции

Результат применения ООП — модульная, расширяемая и переиспользуемая архитектура.

2. Виды моделей информационных систем

Модели ИС — это формализованное представление системы, помогающее понять её структуру, логику и взаимодействия.

Вид моделиЧто описываетПримеры диаграмм
ФункциональнаяПоведение системы, процессы, потоки данныхBPMN, IDEF0, Activity Diagram
ИнформационнаяСтруктуру данных и их связиER‑диаграммы (Chen, Crow’s Foot), UML Class
АрхитектурнаяКомпоненты системы и их взаимодействиеC4 Model, UML Component
ИнтерфейснаяВзаимодействие пользователя с системойWireframes, Mockups, Use Case

Комплексное проектирование требует сочетания всех четырёх видов моделей.

3. Нотации моделирования

3.1. UML (Unified Modeling Language)

Унифицированный язык моделирования, используемый для описания структуры и поведения системы.

  • Use Case Diagram — сценарии взаимодействия акторов с системой
  • Class Diagram — классы, атрибуты, методы, связи
  • Sequence Diagram — последовательность сообщений между объектами
  • Activity Diagram — бизнес‑процессы и логика выполнения
  • Component / Deployment Diagram — архитектура и размещение модулей

3.2. ER‑диаграммы (Entity–Relationship)

Используются для проектирования информационной модели (структуры базы данных). Сущности (прямоугольники), связи (линии/ромбы), атрибуты (овалы).

  • Чена (Chen) — классическая, с атрибутами и связями 1:N, M:N
  • Crow’s Foot — компактная, удобна для СУБД и CASE‑сред
  • IDEF1X — стандартизированная для документирования по ГОСТ 34

3.3. BPMN (Business Process Model and Notation)

Нотация бизнес‑процессов, отражающая логику работы системы и пользователей.

  • Start / End events — начало и завершение процесса
  • Tasks / Sub‑processes — действия
  • Gateways — ветвления (if/else)
  • Flows — потоки сообщений
  • Pools / Lanes — участники процесса

Пример: «Создание заказа» → «Проверка оплаты» → «Отправка уведомления».

3.4. IDEF0 и IDEF3

IDEF0 — функциональная декомпозиция: что система делает (функции, входы, выходы, механизмы, управление). IDEF3 — сценарное моделирование процессов (что и в каком порядке происходит).

Пример для ИС «Театр онлайн»:

  • A0 – Обслуживание зрителей
  • → A1: Приём заказов
  • → A2: Обработка оплаты
  • → A3: Отправка билетов

3.5. C4 Model (Context–Container–Component–Code)

Современная визуальная модель архитектуры ИС (С. Браун) с четырьмя уровнями абстракции.

УровеньНазначениеПример
ContextВзаимодействие системы с внешним окружениемПользователь ↔ Веб‑приложение ↔ СУБД
ContainerКрупные компоненты: UI, API, DB, QueueWebApp, API, DB, EmailService
ComponentВнутренние модули приложенияAuthModule, OrderService
CodeРеализация классов и методовКонтроллеры и репозитории (Flask/Django)

5. Методы нормализации данных и правила формирования ключей

5.1. Нормализация данных

Процесс структурирования таблиц для устранения избыточности и аномалий обновления.

НормаУсловиеПример
1НФКаждый атрибут атомаренПоле «адрес» разделено на город, улицу, дом
2НФКаждое неключевое поле зависит от всего составного ключаТаблица «позиции заказа» содержит только пару order_id + product_id
3НФНет транзитивных зависимостейЦена — в справочнике товаров, а не в заказах

Итог нормализации — минимизация дублирования, упрощение модификаций, логическая целостность данных.

5.2. Правила формирования ключей

  • Первичный ключ (PK): уникально идентифицирует запись (например, user_id).
  • Внешний ключ (FK): связывает таблицы (например, orders.user_id → users.id).
  • Суррогатный ключ: искусственно создаётся (UUID, автоинкремент).
  • Составной ключ: комбинация нескольких полей (например, order_id + product_id).

Хороший ключ — неизменяем, уникален и прост в использовании.

6. Принципы многослойной архитектуры

6.1. Клиент–серверная архитектура

Клиент — пользовательский интерфейс (браузер, приложение). Сервер — логика обработки запросов и доступ к данным. Взаимодействие через HTTP/REST API.

Разделение ответственности Масштабируемость Безопасность

6.2. Модель MVC (Model–View–Controller)

  • Model — данные и бизнес‑логика
  • View — интерфейс пользователя
  • Controller — управление действиями

MVC улучшает читабельность, тестируемость и расширяемость кода.

6.3. Микросервисная архитектура

ПреимуществоСуть
ИзоляцияОтказ одного сервиса не останавливает всю систему
МасштабируемостьУвеличение мощности только нужного модуля
Технологическая свободаКаждый сервис на своём языке и фреймворке
Непрерывная интеграцияЧастые обновления и релизы независимых сервисов

Пример: AuthService → OrderService → PaymentService → EmailService.
